## Context

当前 `workflow.nodes[*]` 仅支持内联 `prompt` 字段，且运行期渲染由自定义正则仅处理 `{{inputs.*}}`。如果新增外部提示词引入能力，继续沿用多阶段/多语法解析会增加普通用户理解和排障成本。该变更目标是把行为收敛到一套模板语义，并明确“极简实现优先”。

## Goals / Non-Goals

**Goals:**
- 支持在单个 `prompt` 中引入多个外部提示词文件。
- 支持 include 与 `{{ inputs.* }}` 在同一节点提示词中混用。
- 使用统一模板引擎处理 include 与变量渲染，消除双阶段语义冲突。
- 使用 Jinja2 经典严格未定义策略（`StrictUndefined`），保证缺失变量快速失败。
- 路径解析固定为“相对配置文件目录”，不受 `cwd` 与 `run.project_root` 影响。
- 内置 preset 与文档示例采用“仅 role 外置、输出协议内联配置”的提示词组织方式，减少提示词碎片化。
- 明确实现边界：仅覆盖普通用户主路径，保持实现极简。

**Non-Goals:**
- 不支持远程 URL、网络下载。
- 不支持旧语法兼容、迁移分支或特判提示（如 `{{prompt_file:...}}` 专门识别）；仅提供文档化手动迁移说明。
- 不新增递归 include 检测、二次渲染、raw 专门处理分支。
- 不为动态/表达式路径参数增加兼容逻辑、静态分析或专门语法拦截分支。
- 不引入固定仓库根、symlink 越界防护等复杂路径安全策略。
- 不改变 `run_workflow` 的节点执行语义（仍是渲染后 prompt 传入 `codex exec`）。

## Decisions

1. 采用 Jinja2 单引擎渲染。
- 规则：`prompt` 统一由 Jinja2 渲染，include 与变量在同一渲染流程完成。
- 规则：Jinja2 使用 `StrictUndefined`，未定义变量保持失败语义，不允许静默空串。
- 规则：保留 `{{ inputs.<key> }}` 作为业务变量入口（与现有 `input_map`/runtime state 语义一致）。
- 规则：`inputs` 的非字符串值保持 JSON 文本注入（保持现有可读性与测试语义）。
- 原因：单一语法降低普通用户心智负担。

2. include 采用 helper 语法 `prompt_file()`。
- 规则：标准写法为 `{{ prompt_file("./prompts/part.md") }}`。
- 规则：一个 `prompt` 可包含多个 include，也可重复引用同一路径。
- 规则：普通用户主路径为“字符串字面量相对路径”。
- 规则：内置 preset 与 README 示例默认仅 include role 文件；输出协议写在节点 `prompt` 内联文本里。
- 规则：实现只做最小运行时校验（参数非空字符串、路径为相对路径），不引入 AST 静态分析。
- 规则：动态/表达式参数不纳入兼容承诺，也不增加专门兼容分支；其行为按运行时结果与通用参数校验处理。
- 原因：函数式语法边界清晰，易于实现与理解。

3. `prompt_file()` 路径解析规则（极简）。
- 规则：路径基准为当前配置文件目录（`Path(config_path).resolve().parent`）。
- 规则：仅允许相对路径；拒绝空路径与绝对路径。
- 规则：允许包含 `..` 的相对路径，按常规 `resolve()` 结果读取；不增加目录越界防护。
- 规则：文件按 UTF-8 读取；不存在或不可读立即报错。
- 原因：覆盖主流场景即可，不为边缘路径策略增加复杂度。

4. 渲染模型采用单次渲染。
- 规则：`prompt_file()` 只负责读取文件并返回原始文本。
- 规则：include 文件内容仅作为文本片段使用。
- 规则：不对引入文本做“再次 include 扫描”或“递归 include 检测”。
- 规则：不对引入文本做二次模板渲染（包括 `inputs` 与任意模板表达式）。
- 规则：include 文件中的模板表达式（如 `{{ ... }}`）按文本保留，不会被再次渲染。
- 规则：该边界在文档中固定声明，不作为后续待决问题反复讨论。
- 规则：模板表达式字面量展示由 Jinja2 原生能力负责（例如 raw 块），本提案不增加额外逻辑。
- 原因：避免实现和测试复杂度膨胀。

5. 错误信息采用最小可执行集合。
- 规则：错误至少包含失败节点提示词字段定位（`node_id` 或 `workflow.nodes[n].prompt` 任一可用）与失败原因。
- 规则：路径相关错误在信息可得时附原始 include 路径；简短修复建议可按可得信息提供。
- 规则：不强制要求 `config_path/config_dir/include_resolved` 等完整结构化字段。
- 规则：在节点渲染时统一包装模板失败（语法错误、未定义变量、`prompt_file` 参数校验、文件读取失败），稳定输出失败节点提示词字段定位（不强制固定格式）。
- 原因：保证可修复即可，避免为报错格式增加维护负担。

6. 测试和文档聚焦主路径。
- 成功：纯内联 `prompt` 继续可运行并保持 `{{ inputs.* }}` 渲染行为。
- 成功：`inputs` 非字符串值在渲染后保持 JSON 文本输出语义。
- 成功：一个 `prompt` 内多个 include + `{{ inputs.* }}` 混用可运行。
- 成功：内置 preset 示例采用“role include + 输出协议内联”的组织方式，兼顾复用与可读性。
- 成功：包含 `..` 的相对 include 路径可按配置目录规则解析。
- 成功：在不同 `cwd` 与不同 `run.project_root` 下，include 解析结果一致。
- 失败：helper 参数非法、绝对路径、文件不存在、不可读、字段类型非法。
- 失败：模板语法错误、未定义变量错误均稳定定位到失败节点提示词字段（`node_id` 或 `workflow.nodes[n].prompt` 任一可用）。
- 失败/边界：动态/表达式 include 参数不纳入兼容承诺，可按运行时结果或普通模板/helper 参数错误处理。
- 回归：include 文件内容不做二次模板渲染（包含 `{{ ... }}` 也按文本保留）。
- 不覆盖：旧语法特判报错、递归 include 专门报错、raw 字面量专门行为测试。
- 文档：聚焦主路径示例（role include + 输出协议内联 + `inputs` 混用），显式写明“include 文件内模板表达式不支持”，并提供 `raw` 块示例用于展示字面量 `{{ ... }}`；补充“旧语法手动迁移”最小清单。

## Implementation Constraints

- 仅考虑普通用户的使用方法，不为极端使用行为引入额外分支。
- 实现优先级：简单、稳定、可维护，而不是“覆盖所有边角情况”。
- 项目当前零用户，不承担兼容包袱，不设计兼容型迁移策略；仅提供文档化手动迁移指引。

## Risks / Trade-offs

- [Risk] 引入文件依赖后，preset 可移植性依赖配置文件相对目录结构
  -> Mitigation: 文档明确“相对配置文件目录”规则并给出目录建议。
- [Risk] 引入 Jinja2 增加一项依赖
  -> Mitigation: 仅启用最小能力（模板渲染 + helper）并补齐主路径测试。
- [Trade-off] 不做递归检测和旧语法特判，错误更“原生”
  -> Mitigation: 文档提供标准写法与常见错误修复示例。
- [Trade-off] 动态/表达式路径参数不纳入兼容承诺，灵活性降低
  -> Mitigation: 先保证普通用户可读、可维护；复杂场景后续再评估是否独立提案扩展。

## Explicitly Ignored In This Change

- 不做旧语法 `{{prompt_file:...}}` 的专门识别或自动迁移。
- 不做 include 文件内递归 include 的专门检测。
- 不为动态/表达式 include 参数增加专门兼容或错误分支。
- 不做复杂路径安全策略扩展（固定仓库根、symlink 越界防护等）。
- 不做“不可读”错误的跨平台细分类（统一按读取失败处理）。
