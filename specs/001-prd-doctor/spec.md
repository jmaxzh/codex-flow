## User Scenarios & Testing *(mandatory)*

### User Story 1 - 全量问题审查 (Priority: P1)

作为需求作者，我希望系统先对需求文档做一次全面审查，并输出可执行的问题清单，以便我明确文档当前的质量缺口。

**Why this priority**: 没有高质量问题清单就无法进入有效修复，审查结果是后续所有动作的输入。

**Independent Test**: 提供一份包含已知缺陷的需求文档，系统可以独立输出完整问题列表（含定位、影响和优先级），即构成可交付价值。

**Acceptance Scenarios**:

1. **Given** 用户提交一份需求文档，**When** 启动审查，**Then** 系统输出结构化问题清单并标记每个问题的严重程度和文档位置。
2. **Given** 用户提交一份已达标文档，**When** 启动审查，**Then** 系统明确返回“无待修复问题”并给出通过结论。

---

### User Story 2 - 问题自动修复 (Priority: P2)

作为需求作者，我希望系统基于问题清单自动修复文档，并给出改动说明，以便快速得到可复审的新版本。

**Why this priority**: 自动修复直接决定效率提升，是该功能区别于普通审查工具的核心价值。

**Independent Test**: 输入原文档与问题清单后，系统可独立产出修复后文档和逐项修复说明，且问题状态发生变化。

**Acceptance Scenarios**:

1. **Given** 存在可修复问题，**When** 执行修复，**Then** 系统产出新文档版本并为每个问题记录“已修复/未修复+原因”。
2. **Given** 某问题缺少必要业务上下文，**When** 执行修复，**Then** 系统保留该问题并明确说明阻塞原因与所需补充信息。

---

### User Story 3 - 循环直到通过 (Priority: P3)

作为需求作者，我希望系统自动执行“审查 -> 修复 -> 复审”的循环，直到文档达到通过标准或达到停止条件。

**Why this priority**: 循环闭环确保质量可收敛，减少人工来回操作。

**Independent Test**: 启动循环后，不依赖外部手工触发，系统能够连续执行多轮并在满足停止条件时输出最终报告。

**Acceptance Scenarios**:

1. **Given** 第一轮修复后仍有未关闭问题，**When** 进入下一轮复审，**Then** 系统继续输出剩余问题并再次执行修复。
2. **Given** 所有必检项均通过，**When** 复审完成，**Then** 系统终止循环并输出“已全部修复”的最终结论。
3. **Given** 达到最大循环轮次且仍有未关闭问题，**When** 系统停止，**Then** 输出未解决问题列表与建议人工处理项。

### Edge Cases

- 文档内容极短、结构缺失或仅有标题时，系统仍需给出可执行反馈，而不是空结果。
- 同类问题在多轮重复出现时，系统按“每轮独立问题集”记录并在轮内避免重复编号，不做跨轮去重。
- 修复动作引入新问题时，系统需要在下一轮显式标记为“新增问题”。
- 文档中存在相互冲突的业务规则时，系统需要标记冲突并阻止“伪修复通过”。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 接收需求文档内容，并支持附带目标读者、业务背景和质量标准等上下文输入。
- **FR-002**: 系统 MUST 在每轮审查中输出结构化问题清单，至少包含问题描述、严重程度、文档定位和影响说明；严重程度枚举固定为 `High`、`Medium`、`Low`。
- **FR-003**: 系统 MUST 为每个问题提供可执行的修复建议，便于后续自动修复或人工处理。
- **FR-004**: 系统 MUST 基于当前未关闭问题生成修复后文档版本，并记录每个问题的处理结果。
- **FR-005**: 系统 MUST 在单轮内维护问题生命周期状态（待处理、已修复、无法修复、人工接受风险）；进入下一轮时重新生成问题编号与状态，不要求跨轮同一问题追踪。
- **FR-006**: 系统 MUST 自动触发“审查 -> 修复 -> 复审”循环，无需用户在轮次间重复手动启动。
- **FR-007**: 系统 MUST 仅在所有必检项通过且未关闭问题数为 0 时结束循环并输出通过结论。
- **FR-008**: 系统 MUST 支持可配置的最大循环轮次（`max_iterations`），默认值为 `20`；达到上限时安全停止并输出未解决问题清单。
- **FR-009**: 系统 MUST 在修复时保持原始业务意图一致，除非问题修复必须调整语义，并需给出调整理由。
- **FR-010**: 系统 MUST 输出最终汇总报告，包含总轮次、各轮发现问题数、各轮修复数、最终轮剩余数和主要风险项。
- **FR-011**: 用户 MUST 能将特定问题标记为“接受风险”，并将该问题状态转为已关闭（接受风险）；后续轮次保留审计记录但不计入未关闭问题数。
- **FR-012**: 系统 MUST 在遇到缺少必要上下文的阻塞问题时，跳过该问题并继续处理其他问题；最终报告需集中输出阻塞原因与所需补充信息。

### Contract & Observability Requirements *(mandatory)*

- **CO-001**: 本特性会变更工作流契约：新增“需求文档专家循环修复”能力，输入包含文档与质量约束，输出包含逐轮问题清单、修复结果和最终结论。相关契约文档需在 `docs/specs/` 中新增或更新，并说明向后兼容策略（未使用该能力的现有流程行为不变）。
- **CO-002**: 必须提供自动化测试证据，覆盖“循环终止条件、单轮内问题状态流转、达到上限后的安全退出、接受风险问题不阻塞结束”等行为；测试资产应落在 `tests/` 下对应编排测试文件中。
- **CO-003**: 运行期观测需在 `.codex-loop-state/` 记录每轮审查与修复摘要，并在最终汇总中新增轮次统计与剩余风险字段；若某字段无变化需显式标注“无变化”。

### Key Entities *(include if feature involves data)*

- **RequirementDocument**: 被审查与修复的需求文档实体，包含当前版本内容、版本号、业务上下文和质量标准。
- **ReviewIssue**: 审查发现的问题实体，包含轮内唯一标识、严重程度（`High`、`Medium`、`Low`）、定位、影响、修复建议和当前状态。
- **IterationRun**: 单轮执行实体，记录轮次编号、输入文档版本、输出问题集、修复结果和耗时。
- **FixDecision**: 对单问题的处理决策实体，标识自动修复、人工接受风险或待补充信息等处理方式。
- **FinalSummary**: 任务结束时的结果实体，聚合总轮次、问题统计、剩余风险和通过结论。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 90% 的标准需求文档可在首次审查中获得完整问题清单，且首轮结果在 2 分钟内可用。
- **SC-002**: 在含已知缺陷的评估样本中，系统在前 2 轮内识别出不少于 90% 的关键缺陷。
- **SC-003**: 对可自动修复的问题，系统在最多 5 轮内的自动关闭率达到 85% 及以上。
- **SC-004**: 至少 80% 的运行任务可在不超过最大轮次的情况下达到“0 未关闭问题”状态并成功结束。
- **SC-005**: 与人工逐条修订流程相比，需求作者完成同等质量文档的平均耗时下降 50% 以上。

## Assumptions

- 输入文档以自然语言正文为主，且至少包含基本章节结构，便于定位与修复。
- 单次任务默认处理一份主需求文档；外部参考资料可作为补充上下文。
- 组织已定义最基本的需求质量标准（如完整性、一致性、可验证性），未提供时使用默认标准。
- 最终对外发布前仍由需求负责人进行人工确认，系统结果作为高质量草稿与审查依据。
- 涉及强监管或法律条款的内容，系统可提示风险但不替代合规审批。
